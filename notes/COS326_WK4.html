type exp =

  | Op of exp * op * exp
  | Var of variable
  | Let of variable * exp * exp
  | Fun of variable * exp
  | App of exp * exp

In this language:

xxxxxxxxxx
Fun (x , y) === fun x -> y
App (Var "fact", Int 3) === fact 3

Note that functions are values, so when we get to a function, we stop evaluation.

Ex: fun x->3 when evaluated is fun x-> 3.

But, fact 3 when evlauted is not fact 3.

How do we evaluate a function application?

xxxxxxxxxx
| App (e1, e2) ->
  (
    match eval e1, eval e2 with
    | Fun (x,e), v2 -> eval (substitute v2 x e)
    | _ -> raise TypeError
  )
  
  (* substitute puts v2 for x in e *)
  
| App (ef, e1) ->
  (
    match eval ef with
    | Fun (x,e2) -> eval (substitute (eval e1) x e2)
    | _ -> raise TypeError
  )
  
  (* this becomes exactly the same as a let expression *)

To see the symmetry between let and function application, observe that these are the same:

xxxxxxxxxx
let x = 1 in x + 41
xxxxxxxxxx
(fun x -> x +41) 1

"Let is syntactic sugar for a function application!"

How about recursive functions?

A let rec definition does two independent things:

  • The rec part allows f to show up in the function body.
  • The let part allows f to show up in the following expression.
xxxxxxxxxx
let rec f x = f( x + 1) in f 3

In the interpreter, we split into two things. A regular let and a new recursive function definition.

xxxxxxxxxx
let f = (rec f x = f (x+1) ) in f 3

This breaks down into two pieces, called the "Principle of Orthogonality".

So, let's add to our definition of an expression:

xxxxxxxxxx
| Rec of variable * variable * exp
(* f *) * (* x *) * (* function body *)

Example:

xxxxxxxxxx
let f = (rec f x = f (x+1)) in f 3
xxxxxxxxxx
Let ("f",
  Rec ("f", "x",
    App (Var "f", Op (Var "x", Plus, Int 1))
  ),
  App (Var "f", Int 3)
)

Actually, a non-recursive function is just a function that doesn't call itself. So, let's delete our Fun of variable * exp variant and only keep Rec of variable * variable * exp.

"Substitute value v for (free occurence of) variable x in expression e:" e[v/x]

Example: (x + y) [7 / y] is (x + 7)

For recursive functions, this becomes:

xxxxxxxxxx
(rec f x = body) arg --> body [rec f x = body / f] [arg/x]  

To mean, that the entire function should be substitute for the function name and the argument should be substituted as the parameter.

When we evaluate this, in OCaml, it looks like this:

xxxxxxxxxx
| App (e1, e2) ->
  Rec (f, x, e) as f_val ->
    let v = eval e2 in
    let body = substitute f_val f (substitute v x e) in
    eval body

Space and Time

Two interesting components:

  • Amount of live space at any instant in time.
  • The rate of allocation, which is asking the question: "Are we constantly discarding and allocating space? Are we putting stress on the garbage collector?". This is not the same as the amount of space used.

Space is taken up by first-order data, function representations, call stacks (which is why tail-recursion is preferred).

Representations:

Emtpy list --> small constant (typically 0).

Non-empty list --> allocated on heap as singly linked list.

Variant type tree = Leaf | Node of int * tree * tree --> small constant for a leaf, else represented as a tag with a pointer to the data carried:

xxxxxxxxxx
Node --> [3, left, right]

Closures

A closure is a pair consisting of the code of a function plus an environment holding the free variables of the function (especially for nested functions).

xxxxxxxxxx
let bar x = x + y

Closure:

xxxxxxxxxx
let f2 n env = 
  n + env.y
(* ***** *)
y = 1

Therefore, a closure has space which is the sum of this pair (usually don't count code though).

Our space estimate will be: two pointers (one to each pair), cost of environment (in this case y = 1), not the cost of the code.

Pruning Closures in Interpreter

You could build a closure environment with all the variables currently in scope.

But, this is overkill, we only need those variables in scope which are also free variables in the function/closure.

There can be asymptotic space difference if you do not obey this.
