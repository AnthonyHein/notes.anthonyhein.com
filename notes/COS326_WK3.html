Define truly new structures that consist of many alternatives with a variant.

type my_bool = Tru | Fal

So that a value of type my_bool is either Tru or Fal.

xxxxxxxxxx
let b1 : my_bool = Tru
let b2 = Fal

This lends itself very nicely into pattern matching:

xxxxxxxxxx
let print_bool (b : my_bool) : unit = 
  match b with
  | Tru -> print_string "True!"
  | Fal -> print_string "False :("

Variant and pattern matching paradigm is nice because the compiler can easily catch non-exhaustive pattern matching.

Variants can carry values aside from tags:

xxxxxxxxxx
type simple_shape =
  | Circle of point * point * point (* center, radius *)
  | Square of point * point * point (* lower left corner, edge *)

See this example:

xxxxxxxxxx
let simple area (s : simple_shape) : float = 
  match s with
  | Circle (_, radius) -> 3.14 *. radius *. radius
  | Square (_, side) -> side *. side

The tag becomes just as important as the values carried.

Type abbreviations can be as simple as type side = float if it improves readability.

Inductive Data Types

A binary tree is:

  • A leaf containing no data.
  • A node containing a key, a value, a left subtree, and a right subtree.
xxxxxxxxxx
type tree =
  | Leaf
  | Node of key * value * tree * tree
x
let rec insert (t : tree) (k : key) (v : value) : tree =
  match t with
  | Leaf -> Node (k , v , Leaf, Leaf)
  | Node (k', v', left, right) ->
    if k < k' then
      Node (k', v', insert left k v, right)
    else if k > k' then
      Node (k', v', left, insert right k v)
    else
      Node (k, v, left, right)

Implementing OCaml in OCaml

Semantics is how the programming language works and comes in these flavors:

  • Operational: how to rewrite program expressions step-by-step until we end up with a value.
  • Denotational: how to interpret a program in a different language that is well understood (like math).
  • Equational: specify an equal program (equal output on all inputs).
  • Axiomatic: provide reasoning rules about programming language.
Implementing an Interpreter

Starts out with a text file (sequence of characters) containing the program source code. Then, parses the sequence of characters and translates it to a data structure that more effectively represents the program. The interpreter evaluates this data structure and makes a new data structure representing the result of evaluation. Optionally, the interpreter prints the result or otherwise communicates to programmer.

Program syntax is a complicated tree-like data structure.

The excess information (like key words, white space) gets purged to create a smaller more digestable Abstract Syntax Tree (AST).

Generally, a let expression has 3 parts, a variable name, a subexpression to be evaluated, and a subexpression to substitute the variable into.

Functional languages like OCaml are terrific for writing compilers because the smooth datatypes (not classes) allow for one datatype per expression.

Example datatype for simple OCaml expressions:

x
type variable = string
type op = Plus | Minus | Times | ...
type exp = 
  | Int of int
  | Op of exp * op * exp
  | Var of variable
  | Let of variable * exp * exp
  
type value = exp
(* 3 + 17 *)
let e = Op (Int 3, Plus, Int 17)
(*
  let x = 30 in
    let y =
     (let z = 3 in z * 4)
    in
    y + y
*)
Let ("x", Int 30,
      Let("y",
            Let("z", Int 3,
                  Op(Var "z", Times, Int 4)),
          Op(Var "y", Plus, Var "y"))

Also, in a tree representation, we can easily identify "free variables" that are not "bound" by some value (note, in recursive function the function name and parameters are bound values).

Do not consider control flow when determining free or bound variables. That is, in if true then x else y both x and y are free variables.

Given a variable occurence, find where it is bound by crawling up AST to closest enclosing let that binds the variable.

In an AST, alpha-conversion is achieved by renaming until old name is bound to another value by another let.

Example: Renaming programmatically using datatype above:

xxxxxxxxxx
let rec rename (x : var) (y : var) (e : exp) : exp =
  match e with
  | Op (e1, op, e2) ->
      Op (rename x y e1, op, rename x y e2)
  | Var z ->
      if z = x then Var y else e
  | Int i ->
      int i
  | Let (z, e1, e2) ->
      Let (z, rename x y e1,
              if z = x then e2 else rename x y e2) (* new variable *)

Next, we write a function which can evaluate an expression.

In a tree, this is seen visually because exp1 will be a value you can substitute in a subexpression.

A value is any unit for which there is no more computation to be done.

Helper functions for this evaluation are eval_op which uses two operands and an operator to get a value and substitute which replaces free instances with a value.

Theorem 1: Well-typed programs have no free variables.

Theorem 2: If e has no free variables and e --> e' then e' has no free variables.

If we are sure that code should never reach a branch in our function, we can raise an exception like this:

xxxxxxxxxx
exception UnboundVariable of string
raise (UnboundVariable "Unbound Variable")

Generally, rule of thumb is that for errors that can and will occur, use options, else use exceptions.
